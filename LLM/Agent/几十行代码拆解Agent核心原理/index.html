几十行代码拆解Agent核心原理

发表于2026-04-26|更新于2026-04-27|LLMAgent

将 AI Agent 拆解为"超级大脑 + 记忆力 + 执行能力"，本文通过四段 Python 代码由浅入深，演示如何从零构建一个简易的Agent。

1. 开篇：Agent 是什么

在聊 Agent 之前，先把它的结构拆解清楚。当前主流范式下，Agent 可以理解为两个核心组件的组合：

Agent=Harness⏟执行框架+LLM⏟推理大脑\text{Agent} = \underbrace{\text{Harness}}_{\text{执行框架}} + \underbrace{\text{LLM}}_{\text{推理大脑}} Agent=执行框架Harness​​+推理大脑LLM​​

组件 名称 职责
LLM 推理大脑 理解用户意图，进行决策和生成
Harness 执行框架 提供记忆管理、工具调用、循环控制等能力

本文接下来的四个 Step，将从最简单的单轮对话，到拥有记忆，再到能够执行命令，最后引入 Skill 规范行为，来说明如何一步步构建一个简易的 AI Agent。

完整代码可详见：https://github.com/doublekehe/mini_agent_demo

问题在哪里？

Agent 此时只能看到当前这一轮对话。"1+1"和"再加1"之间没有任何关联——因为 Agent 没有记忆，每次都是独立的请求。模型不知道"再加1"是在前一句"1+1"的基础上继续提问。

Step 1 的局限：单轮会话，无法跨轮次理解上下文。这就像两个人对话，其中一个人每次开口都失忆。

发生了什么变化？

模型现在能看到完整的对话历史——它知道上一轮回答了"1+1=2"，于是能正确理解"再加1"是在前一个结果上继续。

3.3 上下文记忆 vs 长期记忆

记忆类型 存储位置 生命周期
上下文记忆 messages 列表 仅限当前会话
长期记忆 外部向量数据库、文件 跨会话持久化

当前 Step 2 实现的是上下文记忆——会话结束，记忆消失。真正的长期记忆（如向量数据库检索、RAG 等）是另一个话题，这里先不展开。

4. Step 3：增加执行能力——Execution

4.1 模型能思考，但不会做事

Step 2 的 Agent 虽然能记住对话，但它只能"回复文字"——无法打开文件、执行命令、查询网页。如果用户说"帮我创建一个 hello.txt"，它只能给出一段建议，而不是真正完成这件事。

4.2 让 Agent 输出命令，程序代为执行

解决思路是引入一个 Agent Loop：模型不再是简单的一问一答，而是在一个循环中反复推理，直到认为任务完成。

核心设计：

  • 模型输出固定格式的指令：命令:xxx 或 完成:xxx
  • 程序检测输出前缀：若是"命令"，则调用 os.popen 执行；若是"完成"，则退出循环

4.3 Agent Loop 详解

图：Agent Loop 执行循环——模型推理 → 判断是否需要执行命令 → 系统执行 → 结果回传 → 再次推理，直到输出"完成:xxx"退出循环。

4.4 "命令:"和"完成:"的契约

这是一种结构化输出契约——通过固定前缀约定，让模型和程序之间建立了一套简明的通信协议：

模型输出 程序行为 退出循环？
命令:touch hello.txt 执行 touch hello.txt 继续循环
完成:文件创建成功 打印结果 ✅ 退出

这也是 OpenClaw 中 Harness 框架的核心思想：LLM 负责推理决策，Harness 负责执行和循环控制，两者各司其职。

5. Step 4：引入 Skill 规范行为——Skill

5.1 直接执行命令，效果往往不够好

Step 3 让 Agent 拥有了执行能力，但有一个明显问题：模型给出的命令质量不稳定。

以"下载 Transformer 论文"为例，模型可能会：

  • 输出 wget xxx 但不指定文件名，导致文件以 URL 哈希命名，难以查找
  • 输出 curl xxx 但忘记加 -o 参数，文件直接打印到终端
  • 不知道用哪个工具（wget vs curl vs python -c "..."）

解决方案是把"做某类事的标准流程"打包成一段提示词（Skill），在系统级别注入给模型：

5.3 Skill 的本质

要素 说明
是什么 一段结构化的指令，告诉模型"遇到 X 类情况时，应该怎么做"
解决什么问题 模型行为不稳定、执行命令不规范
和 System Prompt 的区别 System Prompt 是通用原则；Skill 是垂直场景的专业 SOP

6. 总结与展望

6.1 四阶段演进路径

每一步都在前一步基础上增加一个维度，最终组合成一个真正可用的 Agent。

6.2 Harness 的本质

回顾开篇的定义：

Agent=Harness⏟执行框架+LLM⏟推理大脑\text{Agent} = \underbrace{\text{Harness}}_{\text{执行框架}} + \underbrace{\text{LLM}}_{\text{推理大脑}} Agent=执行框架Harness​​+推理大脑LLM​​

Harness 负责的事情包括：

  • 循环控制：管理 Agent Loop 的启动、退出、重试
  • 记忆管理：维护上下文消息、长期知识存储
  • 工具调用：执行命令、调用 API、操作文件
  • Skill 注入：为模型注入专业知识和行为规范

LLM 本身只是一个"推理引擎"——它能思考，但不会行动。Harness 赋予了它记忆和手脚，让它成为一个真正的 Agent。

6.3 未来方向

本文演示的是最简化的 Agent 原型，工业级 Agent 还需要：

  • 多工具调用：不仅能执行 Shell，还能调用 API、读写数据库
  • 规划能力：将复杂任务拆解为子步骤，逐一完成（ReAct、Plan-and-Execute 范式）
  • 长期记忆：向量数据库 + RAG，跨会话持久化知识
  • 安全沙箱：命令执行在隔离环境中，防止恶意操作

理解了这四步的演进逻辑，再去看 OpenClaw、Claude Code 等框架，就会发现它们的本质都是同一个：Harness + LLM = Agent，只是 Harness 的能力边界不同。

一句话总结：Agent 不是魔法，它的每一种能力都可以拆解——对话来自 LLM，记忆来自 Harness，执行命令来自 Agent Loop，规范行为来自 Skill。理解了这四步，就理解了所有 Agent 的底层逻辑。
