读懂FLOPs与算力利用率

发表于2026-07-12|更新于2026-07-12|LLM模型推理

模型训练效率里最容易混在一起的，是“模型要算多少”“硬件实际算了多少”“峰值算力有多少转化成模型进度”和“GPU 到底忙不忙”。FLOPs、HFU、MFU 和 GPU 利用率分别回答这些问题。

1. FLOP：一次浮点运算

FLOP 是 Floating-Point Operation 的缩写，表示一次浮点运算。这个缩写通常理解为从 FLoating-point 取 FL，从 OPeration 取 OP，合成 FLOP。浮点数是计算机表示实数的一种方式，例如 3.14、-0.125 和 1.2e-5。一次浮点加法或乘法通常各记作 1 FLOP。

这里有三个容易混淆的写法：

写法 常见含义 回答的问题
FLOP 一次浮点运算 一次操作是什么
FLOPs 浮点运算总次数 这项任务要算多少
FLOP/s 每秒浮点运算次数 设备算得多快

注意：用 FLOPs 表示总计算量，用 FLOP/s 表示计算速度。下面我们用一个矩阵乘法的例子来说明如何计算一个任务有多少FLOPs。

1.1 矩阵乘法有多少 FLOPs

设有两个矩阵：

A∈Rm×k,B∈Rk×nA \in \mathbb{R}^{m \times k}, \qquad B \in \mathbb{R}^{k \times n} A∈Rm×k,B∈Rk×n

矩阵乘法得到：

C=AB,C∈Rm×nC = AB, \qquad C \in \mathbb{R}^{m \times n} C=AB,C∈Rm×n

CCC 一共需要通过 m×nm \times nm×n 次 点积 计算得到，每次点积运算需要 kkk 次乘法和 k−1k-1k−1 次加法，故矩阵乘法需要的运算次数为：

FLOPsmatmul=mn(2k−1)\text{FLOPs}_{\text{matmul}} = mn(2k-1) FLOPsmatmul​=mn(2k−1)

工程估算更常使用近似口径，把一次乘加看作 2 FLOPs：

FLOPsmatmul≈2mkn\text{FLOPs}_{\text{matmul}} \approx 2mkn FLOPsmatmul​≈2mkn

图 1：矩阵乘法的 FLOPs。示例中 A 为 3×4，B 为 4×3，输出 C 为 3×3

神经网络里的 Linear、Attention 和 FFN 等，底层都包含大量矩阵乘法。矩阵乘法的 2mkn2mkn2mkn 估算，会反复出现在模型 FLOPs 的计算里。后面分析 MFU 时，也是在这个约定口径下讨论“模型理论上需要算多少”。

2. MFU：模型有效计算占了多少峰值算力

MFU（Model FLOPs Utilization）即模型 FLOPs 利用率，表示模型有效计算吞吐占全部 GPU 理论峰值吞吐的比例。MFU 的分子不是硬件实际执行过的全部计算量，而是按模型结构估算的理论有效 FLOPs，通信、等待、数据加载不增加模型有效 FLOPs，却会消耗训练时间，因此会拉低 MFU。

2.1 MFU 的计算

MFU 的计算可以统一写成：

MFU=Funit×RPtotal\text{MFU} = \frac{F_{unit} \times R} {P_{total}} MFU=Ptotal​Funit​×R​

其中，FunitF_{unit}Funit​ 是完成一个训练单位所需的模型 FLOPs，RRR 是每秒完成的训练单位数，PtotalP_{total}Ptotal​ 是全部 GPU 的理论峰值 FLOP/s。训练单位可以是 token、sample 或 step，但 FunitF_{unit}Funit​ 和 RRR 的单位必须一致。例如，使用每个 token 的 FLOPs 时，吞吐就要使用 tokens/s。

token、sample 和 step 描述的是不同粒度的训练工作：

训练单位 含义 对应吞吐 使用时要注意
token 文本经过 tokenizer 切分后的基本单位 tokens/s 最适合语言模型；一个 sample 往往包含多个 token
sample 一条训练样本，例如一段文本或一组问答 samples/s 不同 sample 的 token 数可能不同，单看 samples/s 容易忽略长度差异
step 训练日志中的一次迭代计数 step/s 它指一次 micro-batch

模型 FLOPs 表示完成一个训练单位理论上需要多少浮点运算，由模型结构和统计口径决定，不是 GPU 实际执行的指令数。

训练吞吐 表示单位时间完成多少训练工作，例如 tokens/s、samples/s 或 step/s。

GPU 峰值 FLOP/s 表示硬件理论上每秒最多能完成多少浮点运算。若使用 GGG 张相同 GPU，单卡峰值为 PpeakP_{peak}Ppeak​，则总峰值为：$ P_{total}=G \times P_{peak} $。

假设完成一个 step 模型需要进行运算 100 TFLOPs，实测吞吐为 2 step/s，全部 GPU 的理论峰值为 1000 TFLOP/s，则：

MFU=100×21000=20%\text{MFU} = \frac{100 \times 2}{1000} =20\% MFU=1000100×2​=20%

这表示全部 GPU 理论峰值算力中，有 20% 转化成了模型有效计算。

图 2：MFU 等于模型有效 FLOP/s 除以硬件峰值 FLOP/s；其中，模型有效 FLOP/s 由每个 step 的模型计算量乘以实测 step 吞吐得到。

3. HFU：硬件实际发挥了多少算力

HFU 是 Hardware FLOPs Utilization，即硬件 FLOPs 利用率。它表示 GPU 实际执行的浮点运算吞吐，占理论峰值吞吐的比例：

HFU=Fhardware/tPtotal\text{HFU} = \frac{F_{hardware}/t} {P_{total}} HFU=Ptotal​Fhardware​/t​

其中，FhardwareF_{hardware}Fhardware​ 是一段时间内硬件实际执行的浮点运算量，单位是 FLOPs；ttt 是这段计算的真实耗时；PtotalP_{total}Ptotal​ 是全部 GPU 的理论峰值 FLOP/s。

假设 GPU 在 0.5 秒内实际执行了 300 TFLOPs，而 GPU 的理论峰值为 1000 TFLOP/s，那么：

HFU=300/0.51000=60%\text{HFU} = \frac{300/0.5}{1000} =60\% HFU=1000300/0.5​=60%

这表示硬件实际浮点运算吞吐达到了理论峰值的 60%。

HFU 和 MFU 的区别在分子，MFU 统计模型定义中的有效计算，HFU 统计硬件实际执行的全部浮点运算。Activation Checkpointing 是一个典型例子：它在反向传播时重新计算部分前向结果，这些重计算会消耗硬件算力，因此计入 HFU；但它们没有增加模型本身的有效训练工作，通常不计入 MFU。在这种情况下，HFU 会高于 MFU。

三个指标可以这样区分：

指标 统计口径 主要回答的问题
GPU-Util GPU 执行 kernel 的时间比例 GPU 忙不忙
HFU 硬件实际执行的 FLOP/s ÷ 理论峰值 FLOP/s 计算硬件发挥了多少峰值能力
MFU 模型有效 FLOP/s ÷ 理论峰值 FLOP/s 峰值算力有多少转化成模型进度

4. GPU 利用率：GPU 有多少时间在忙

我们常说的 GPU 利用率，通常指 nvidia-smi 中的 GPU-Util。它大致表示：在最近一个采样周期内，有一个或多个 kernel 在 GPU 上执行的时间比例。

如果采样窗口内 GPU 始终有 kernel 在运行，GPU-Util 可以接近 100%。但这个指标不会告诉我们：

  • kernel 使用了多少个计算单元；
  • 执行的是高吞吐矩阵乘法，还是低算术强度的逐元素操作；
  • 多张 GPU 是否在等待通信或同步。

因此，GPU-Util 为 100% 只能说明“GPU 基本没有完全闲下来”，不能说明“GPU 已达到峰值算力”。

图 3：GPU 利用率关注忙碌时间，MFU 关注模型有效计算吞吐，两者可以同时很高，也可以明显背离。

GPU-Util 高、MFU 低，通常说明时间花在了“让 GPU 忙起来”的事情上，但没有持续产出高吞吐的模型 FLOPs。常见原因有几类：

  • 矩阵太小：batch size、sequence length 或 hidden size 太小时，kernel 虽然在运行，却没有足够的并行任务填满所有计算单元。
  • 通信和同步占用时间：数据并行、Tensor Parallel、Pipeline Parallel、FSDP/ZeRO 都会引入不同形式的通信或等待。通信操作可能让 GPU 看起来活跃，但它们不增加模型理论 FLOPs。
  • 数据加载形成空档：dataloader、tokenization、数据解压或远程存储跟不上时，GPU 会在 step 之间等待。
图 4：只有一部分 step 时间用于高吞吐矩阵计算，其余时间可能花在访存、通信、同步和数据等待上。

5. 总结

几个概念可以压缩成四句话：

  1. FLOPs 描述完成模型计算理论上需要做多少浮点运算。
  2. MFU 描述模型有效 FLOP/s 占 GPU 理论峰值 FLOP/s 的比例。
  3. HFU 描述硬件实际执行的 FLOP/s 占理论峰值 FLOP/s 的比例。
  4. GPU 利用率 描述 GPU 有多少时间在执行 kernel，不等于算力利用率。

GPU 利用率接近 100% 而 MFU 不高，并不矛盾。GPU 可能一直在执行小 kernel、搬运数据或参与通信，却没有持续运行高吞吐矩阵计算。
