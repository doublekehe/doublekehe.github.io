2.1 根源：浮点数的表示误差

计算机底层用 IEEE 754 双精度浮点数（64-bit） 存储小数。一个浮点数由三部分组成：

部分 位数 作用
符号位 S 1 bit 正负
指数位 E 11 bits 决定数量级
尾数位 M 52 bits 决定精度

十进制分数 110\frac{1}{10}101​（即 0.1）转换为二进制是无限循环的：

110=0.0001100110011001100110011001100110011…(2)\frac{1}{10} = 0.0001100110011001100110011001100110011\ldots_{(2)} 101​=0.0001100110011001100110011001100110011…(2)​

52 位的尾数有限，无限循环的小数必须被截断。 这就引入了表示误差——我们存储的 0.1 实际上只是真实值的一个近似。

所以 0.1 + 0.2 的结果并不是精确的 0.3，而是多出了 0.00000000000000004 左右。

浮点数的 == 比较，本质上要求两个值的 IEEE 754 位模式完全相同。只要存在任何微小的差异，结果就是 False。

2. 标准方案：math.isclose()

参数 含义 默认值
rel_tol 相对误差容差 1e-09
abs_tol 绝对误差容差 0.0

比较逻辑：

∣a−b∣≤max⁡(rel_tol×max⁡(∣a∣,∣b∣),  abs_tol)|a - b| \leq \max(\text{rel\_tol} \times \max(|a|, |b|),\; \text{abs\_tol}) ∣a−b∣≤max(rel_tol×max(∣a∣,∣b∣),abs_tol)

2.1 使用示例

比较方式 0.1+0.2 vs 0.3 sqrt(2)² vs 2.0 1e8 vs 1e8+1e-8
== False ❌ False ❌ False ❌
isclose True ✅ True ✅ True ✅

3. 动手实现：自定义浮点相等判断

math.isclose 的核心思想是：误差应该和数值的大小成正比。大的数量级允许更大的绝对误差，小的数量级则需要更精确的比较。

将比较条件拆解：

  • 相对误差 ∣a−b∣≤rel_tol×max⁡(∣a∣,∣b∣)|a - b| \leq \text{rel\_tol} \times \max(|a|, |b|)∣a−b∣≤rel_tol×max(∣a∣,∣b∣)
  • 绝对误差 ∣a−b∣≤abs_tol|a - b| \leq \text{abs\_tol}∣a−b∣≤abs_tol

两者取最大值，是为了同时照顾到大数和小数。

4. 核心结论

  • 不能用 == 比较浮点数，因为浮点数的表示误差使得"看起来相等"的值实际上有不同的二进制位模式
  • math.isclose() 是标准方案，同时考虑相对误差和绝对误差，适用于绝大多数场景
  • 金融/金额场景不要用浮点数，用 decimal.Decimal 或整数（以"分"为单位）完全避免这个问题

浮点相等比较的本质不是判断"是否相同"，而是判断"是否足够接近"——用误差容差代替精确比较，才是正确思路。
