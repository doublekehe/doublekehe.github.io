搜推模型如何Scaling

发表于2026-07-20|更新于2026-07-20|推荐算法排序Scaling

排序模型 scaling up，是让搜索与推荐中的排序模型像 LLM scaling law 一样：随着有效参数量、训练计算量和数据规模增加，模型效果仍能稳定变好。这篇文章重点介绍搜推排序模型如何从 Embedding + MLP 范式走向 Embedding + Tokenization + Transformer-like Backbone 新架构，从而实现模型 Scaling Up。

1. 搜推系统基本架构

一个完整的工业界搜索或推荐系统，因为ms级的RT限制，无法实现全库打分，通常会拆分为底池 → 召回 → 粗排 → 精排 → 重排几个阶段 ，整体形状像一个倒漏斗。

  • 底池（Corpus）：存放所有可被检索或分发的商品、视频、文档、音乐、广告、POI 等内容，规模通常在百万到十亿级。底池决定了系统“有什么可以推荐”，通常还会通过上下架、地域、权限和安全规则完成最基础的过滤。
  • 召回（Retrieval）：从全量底池中快速找回数千到数万条可能相关的候选。这个阶段更看重召回率，通常并行使用协同过滤、向量检索、规则召回、图召回等多条通道，尽量避免有价值的物品过早漏掉。
  • 粗排（Pre-Rank）：用计算成本较低的模型对召回结果做第一次统一比较，把候选进一步压缩到百级或千级。它需要在速度和精度之间取平衡：既不能像召回一样只做近似匹配，也承担不起精排模型的完整计算。
  • 精排（Ranking）：对留下来的候选逐一做更准确的偏好预测。模型会综合用户、物品、查询、上下文、历史行为和交叉特征，输出 CTR、CVR、GMV、观看时长、完播率等分数，再据此形成排序结果。
  • 重排（Re-Rank）：在主排序分数之上做列表级决策，处理多样性、新鲜度、去重、探索、业务规则和生态约束，最终选出展示给用户的几十条结果。
图 1：搜推系统多阶段级联架构，精排综合 用户、物品和上下文特征完成打分。

在整条搜推链路中，排序模型使用的特征最丰富、模型结构最复杂，对最终效果的影响也最大。它需要综合 Query、用户、物品、行为序列与上下文信息，对候选物品逐一预估 CTR、CVR、GMV、观看时长等目标，其预测精度会直接影响点击、转化、时长和收入。

因此，如何持续提升排序模型的能力，一直是搜索与推荐系统优化的核心问题。本文重点讨论的正是排序模型的 scaling up：如何在严格的线上时延约束下扩大模型的有效参数量和计算量，并让模型效果随着规模增长继续稳定提升。

2. 排序模型的任务与技术演进

在具体讨论排序模型如何做scaling之前，我们先回顾一下排序模型的基本任务和技术演进，看看排序模型是如何从简单的UserCF、LR，逐步发展到深度学习、序列建模，再走向scaling的。

2.1 排序模型的基本任务：用户 x 商品 → score

经过召回和粗排后，系统已经从海量底池中筛出了数百到数千个候选物品。排序模型的基本任务，就是结合当前用户、候选物品、Query、上下文和用户历史行为，对每个候选物品逐一预测用户发生点击、转化、观看等行为的可能性或价值，再根据这些预测结果完成排序。

对用户 uuu 和候选物品 iii，可以把这个任务写成：

y^u,i=fθ(u,i,q,c),\hat{\boldsymbol y}_{u,i}=f_\theta(u,i,q,c), y^​u,i​=fθ​(u,i,q,c),

其中，qqq 是搜索场景中的 Query，ccc 是时间、位置、入口等上下文。

不同业务场景下，用户会产生不同的关键动作，排序模型需要针对这些动作预测相应的发生概率或期望价值：

表 2-1　不同业务场景中的关键用户动作与模型预测目标
业务场景 关键用户动作 模型预测目标
电商 点击、下单、支付金额 CTR、CVR、CTCVR、成交GMV
音乐 听完、收藏 完播率、播放时长、收藏率
短视频 点击、观看、完播、点赞、关注 CTR、预期观看时长、完播率、点赞率、关注率
本地生活 点击、到店、下单、核销 CTR、到店率、CVR、核销率、预期交易金额
注：同一业务通常采用多任务学习同时预测多个目标。概率类目标用于衡量动作发生的可能性，时长与金额类目标用于衡量用户行为产生的预期价值；实际排序分数还会结合业务目标进行加权或约束。

排序模型基础结构包含三部分：

  • 输入用户、物品、上下文和行为序列等相关特征，搜索场景还会输入 Query 特征；
  • 中间由排序模型完成从特征到目标的映射；
  • 最后输出 CTR、CVR、完播率、观看时长等预测值。
图 2：排序模型的基础框架。

排序模型可以是 LR、GBDT，也可以是 DNN、Transformer-like的模型。但无论内部结构如何变化，排序模型的基本任务始终没有改变：预估当前用户对每个候选物品上的潜在价值。模型的演进，也主要围绕着 特征如何表示、不同特征如何交叉、行为序列做兴趣建模，以及模型如何Scaling，从更多参数和计算中 持续获益。沿着这条主线，下面回顾排序模型如何从协同过滤与浅层模型，逐步走向深度学习、序列建模与 Scaling 时代。

2.2 排序模型的五个演进阶段

回顾推荐与排序模型的发展，不断引入新的信息和新的建模方式，其结构演进大致可以概括为五个代表阶段。

第一阶段是协同过滤与矩阵分解。 这一阶段核心解决的问题是：在用户只与少量物品发生交互的情况下，如何利用群体行为中的协同信号，推断用户对未交互物品的偏好。 Herlocker 等人在 1999 年将 UserCF 从早期系统实践整理为较完整的邻域协同过滤算法框架：先计算目标用户与其他用户的相似度，再选择一组近邻，最后对近邻评分加权聚合得到预测；论文还系统比较了相似度、近邻选择和权重修正等组件。ItemCF 沿用相同的邻域思路，只是将相似性建模对象从用户换成物品。矩阵分解则进一步把稀疏的用户—物品交互矩阵 RRR 近似分解为两个低秩矩阵 PPP 和 QQQ：每个用户对应一个隐向量 pup_upu​，每个物品对应一个隐向量 qiq_iqi​，两者的内积 puTqip_u^\mathsf{T}q_ipuT​qi​ 就是用户对物品的基础偏好分数。2006 年 Netflix Prize 发布大规模用户评分数据后，Simon Funk 公开了使用随机梯度下降、仅针对已观测评分训练正则化隐因子的方法，后来通常被称为 FunkSVD。竞赛推动矩阵分解在偏置建模、隐式反馈、时间动态和模型融合等方向快速演进；Koren、Bell 和 Volinsky 在 2009 年对这一范式进行了系统总结。矩阵分解由此能够利用低秩结构在稀疏交互之间共享统计信息，获得更紧凑、泛化性更好的用户和物品表示。
但这一阶段学习的主要依据仍是用户 ID、物品 ID 及其交互关系：没有历史行为的新用户、新物品难以得到可靠表示，也不容易直接利用内容、Query、上下文和行为时序等信息。

第二阶段是特征工程与浅层排序。 这一阶段核心解决的问题是：如何突破仅依赖用户 ID、物品 ID 和历史交互的局限，把用户、物品、Query 和上下文等多源业务信息引入排序，并表达特征之间的组合关系。 随着搜索广告和工业推荐发展，排序模型开始显式使用用户、物品、Query 和上下文特征。LR 对一阶特征进行线性加权；POLY2 为每一对特征组合学习独立权重，显式建模二阶交互，但参数量随特征规模快速增长，并且难以估计训练数据中未出现过的稀疏组合；FM 则用两个特征隐向量的内积表示交互权重，在大幅压缩参数的同时，让稀疏特征之间能够共享统计信息。另一条路线由树模型推动：GBDT 学习非线性划分，GBDT+LR 再把树的叶节点转化为稀疏组合特征交给 LR 预测。模型由此获得了比单纯 LR 更强的特征交互能力，但整体效果仍然高度依赖特征设计，能够表达的交互阶数和结构也受到模型形式限制。

图 3：排序模型的五个演进阶段

第三阶段是深度特征表示与交互。 这一阶段核心解决的问题是：如何减少对人工特征交叉的依赖，让模型从大规模稀疏特征中自动学习低维表示以及复杂的高阶交互。 2016 年前后，Embedding + DNN 开始成为工业排序模型的基础范式。Wide & Deep（W&D，也常简称 WDL）将 Wide 侧对高频共现的记忆能力与 Deep 侧的泛化能力结合起来；Deep Crossing 用 Embedding、Stacking Layer 和残差网络隐式学习高阶特征组合。随后，DeepFM 在 2017 年让 FM 与 DNN 共享 Embedding，在同一模型中分别学习低阶和高阶交互；DCN 使用 Cross Network 显式、逐层学习有界阶交互；xDeepFM 又在 2018 年通过 CIN 显式建模向量级高阶交互，并与 DNN 的隐式交互能力结合。排序模型由此开始自动学习稀疏 ID 的表示和复杂特征组合，显著减少对人工交叉特征的依赖。不过，这些模型的主干仍以固定长度特征向量为中心，后续提升往往依赖继续增加新的交互模块。

第四阶段是行为序列与兴趣建模。 这一阶段核心解决的问题是：如何突破单一静态用户向量的表达限制，从历史行为序列中刻画不断变化、多样化、长期且与当前候选相关的用户兴趣。 这一阶段围绕序列加宽、序列加长、多兴趣和不同序列建模方式四个方向展开。序列加宽，为每个行为加入类目、品牌、动作类型、时间间隔、位置、场景与多模态信息；序列加长，从近期几十次行为扩展到千级、万级乃至全生命周期；多兴趣建模，将单一固定用户向量扩展为候选感知的兴趣表示或多个兴趣向量；序列建模方式，则从 Avg Pooling 逐步走向 Target Attention、Self-Attention/Transformer，以及面向超长序列的检索、采样与压缩等。比如，YouTube DNN 将用户历史观看/行为过的物品 Embedding 后做 Average Pooling，是工业界利用行为序列的重要起点。DIN 通过target attention 用候选物品激活相关行为，BST 进一步用位置特征和 Transformer 建模行为间依赖。MIND 通过多个兴趣向量服务召回，DMIN 则将多兴趣思想用于 CTR 预估。SIM、ETA 和 TWIN 继续向超长序列推进，分别探索两阶段检索、端到端哈希检索和两阶段一致性；LONGER 又将长序列 Transformer、Token Merge 与 GPU 系统优化结合起来。逐步把用户表示从静态画像推进为更丰富、更长、更细粒度且与候选相关的动态兴趣。

第五阶段是 Token 化与 Transformer-like 主干 Scaling。 这一阶段核心解决的问题是：如何摆脱不断堆叠定制模块所带来的结构碎片化，构建统一、可重复堆叠且硬件友好的模型主干，让更多 Token、参数和计算能够更稳定地转化为模型效果。 传统排序模型通常将特征 Embedding 后直接拼接或池化，再交给 MLP 和各类定制交互模块；模块越叠越多，结构越容易碎片化，新增参数和计算也未必能稳定转化为效果。新范式先将用户、物品、上下文和行为序列等异构特征组织成结构化 Token，再交给可重复堆叠的 Transformer-like Backbone 统一建模，也就是从 Embedding + MLP 升级为 Embedding + Tokenization + Transformer-like Backbone。这里的 Transformer-like 并不等于照搬标准 Transformer，关键是形成“结构化 Token + 可重复堆叠 Block”的扩展接口，使模型能够沿深度、宽度、Token 数、序列长度等维度扩展。Wukong、HSTU、RankMixer、OneTrans、UniMixer 和 TokenMixer-Large 等工作，则从不同角度改造 Tokenization、Attention、FFN、Norm 与 Residual 等模块，以可重复堆叠的 Block 统一特征交互与序列建模。

当前，搜推排序模型已进入到 基于Transformer-like 主干 实现Scaling 的阶段。接下来，我们重点讨论搜推排序模型如何实现 Scaling Up。

3. 搜推排序模型如何Scaling

随着用户和物品规模、特征数量、行为序列长度和业务目标不断增加，搜推任务变得越来越复杂，在Embedding + MLP 的范式下，通常会外挂大量特征交叉、多种序列建模方式和多个辅助任务，结构偏碎片化，存在大量Concat/Pooling/小矩阵运算，MFU较低（通常<10%），而LLM的MFU通常在30%~50%或更高。能否借鉴 LLM 的 Scaling 思路，构建统一、可扩展且硬件友好的模型主干，使更多有效参数和计算持续转化为效果提升，是搜推模型 Scaling Up 的核心目标。

3.1 搜推 Scaling Up 的两条路线：生成式模型与判别式模型

对于搜推模型如何Scaling，有2个主流的路线：

  • 1）生成式模型才能更好地做scaling，将搜推改造成NTP任务，直接生成下一物品序列，典型工作如快手的OneRec/OneSearch系列；
  • 2）判别式模型也能做Scaling，保留现有搜推多阶段级联结构，主要聚焦于扩展模型的有效参数和计算量，典型工作如字节的rankmxier系列。

注：生成式路线尝试将搜推任务改写为序列生成问题，并可能进一步统一召回与排序；判别式路线保留现有多阶段级联链路，重点扩展精排模型的有效参数和计算量。两条路线并非完全互斥，实际系统也可能采用混合架构。

3.2 判别式搜推模型 Scaling 的基础范式

判别式搜推模型通过借鉴 LLM 统一输入表示、可堆叠主干和持续扩展有效计算的方式，将原本不断在 Embedding + MLP 上堆叠碎片化结构的范式，升级为规整、可重复堆叠的 Embedding + Tokenization + Transformer-like 结构。

图 7：两种判别式排序模型基础范式

在新的建模范式下，扩容方式从“继续外挂序列建模模块、CrossNet 或手工交叉模块”，转变为增加 Token 数量和维度、Block 层数、主干宽度或专家容量。要让这套范式真正适用于搜推排序，核心需要解决两个问题。

问题一：异构特征如何 Token 化。 LLM 的原始输入是文本，文本天然具有顺序结构，并且可以通过统一的词表和 Tokenizer 映射为统一的离散 Token 序列。搜推排序的输入则同时包含用户与物品 ID、数值统计、Query、上下文、交叉特征和变长行为序列。这些特征的来源、语义、粒度和时序属性都不相同，无法像文本一样直接组成统一的 Token 序列。因此，需要设计适合搜推特征的 Tokenizer：对异构特征进行分组、切分与投影，对齐到统一维度，并注入类型、位置、时间等信息，最终形成 X∈RT×DX\in\mathbb R^{T\times D}X∈RT×D 的结构化 Token 表示。其中，TTT 是 Token 数，DDD 是统一的 Token 维度。

问题二：如何构建可扩展且高效的 Transformer-like 主干。 即使完成了 Tokenization，也不能直接把标准 Transformer 搬到线上精排。Self-Attention 的计算复杂度约为 O(T2D)O(T^2D)O(T2D)，随着 Token 数量和行为序列长度增加，计算量与时延会快速增长；而线上排序同时受到严格的 RT 和成本约束。因此，新的主干一方面能够通过增加深度、宽度或专家容量持续扩展模型能力，另一方面也要简化 Self-Attention 的计算，让新增参数能够高效执行，在满足线上 RT 的前提下转化为效果收益。

图 8：新建模范式需要解决的两个核心问题

围绕这两个问题，不同工作对异构特征的组织粒度、Token 生成方式以及 Transformer-like 主干的结构选择各不相同，接下来将通过相关代表性工作，进一步梳理这两条主线上的主要设计思路。

4. 总结与实践建议

前面从技术演进、Scaling 路线、基础范式和代表工作几个层面，介绍了搜推排序模型从 Embedding + MLP 走向 Embedding + Tokenization + Transformer-like Backbone 的过程。总体来看，新范式主要围绕两个问题展开：输入侧需要将异构特征组织为结构化 Token，主干侧需要在严格的线上约束下持续扩展有效容量。

4.1 异构特征 Tokenization

代表工作的第一个分歧，是如何确定 Token 的语义边界和粒度。常见做法可以归纳为：

表 4-1　异构特征 Tokenization 的常见方法及权衡
方法 核心做法 优点 主要风险
Field-wise 每个 Field 对应一个 Token 字段边界清楚，结构最直观 Token 数量过多，计算成本较高
Semantic group-wise 将用户、物品、序列、交叉等同类特征组内拼接后投影 兼顾语义边界与计算效率 分组依赖业务经验，组内信息可能被压缩
Sequence item-wise 每个行为 Item 对应一个 Token，并叠加动作、时间和位置信息 保留完整时序，适合序列建模 长行为序列会显著增加 Token 数
Domain/type-wise 按用户、物品、Query、行为序列等 Domain 分组，并采用不同投影参数 能够保留不同特征空间的差异 参数与工程实现更复杂
Global/summary token 增加 Global Token 汇总全局信息 便于不同特征组之间传递信息 全局 Token 可能成为信息瓶颈
注：上述方法并非互斥，同一个模型通常会组合使用多种 Tokenization。例如，非序列特征可以按语义或 Domain 分组，行为序列可以按 Item 展开，同时增加 Global Token 汇总全局信息。实际选择需要在业务语义保留、Token 数量、主干计算量和工程复杂度之间权衡。

Tokenization 不只是数据预处理，还决定了后续主干看到的信息结构。除了分组粒度，还需要明确数值特征如何投影、多值 ID 如何聚合、行为序列是否按 Item 展开、时间和动作类型如何注入，以及不同 Token 是否共享投影参数。

4.2 Transformer-like 主干

标准 Transformer Block 主要由 Self-Attention、FFN、Norm 和 Residual 组成。搜推模型的结构创新，也可以沿着这几个部分理解。

1）Self-Attention / Token 间交互。 标准 Self-Attention 能够根据当前样本动态计算 Token 之间的交互权重，但复杂度约为 O(T2D)O(T^2D)O(T2D)。面向线上排序，不同工作采用了不同的简化方式：HSTU 使用推荐特化的 Pointwise Aggregated Attention；OneTrans 使用 Mixed Causal Attention、序列压缩和 KV Cache；RankMixer、TokenMixer-Large 使用规则化的重排结构；UniMixer 则进一步把规则交互参数化。除此之外，还可以采用分块、局部、稀疏、低秩等方式降低计算量。

2）FFN / 容量扩展。 FFN 承担每个 Token 内部的非线性变换，也是增加稠密参数最直接的位置。标准 Transformer 对所有 Token 共享同一套 FFN，但搜推 Token 的语义高度异构，因此出现了 Per-token FFN、Mixed FFN 和 Token-specific FFN。进一步引入 Sparse MoE 后，可以通过增加专家数扩大模型总容量，同时只激活少量专家以控制单次推理 FLOPs。

3）Norm 与 Residual / 深层稳定性。 当主干从少量 Block 扩展到更深结构时，Norm 和 Residual 决定新增层数能否被充分训练。RMSNorm、SiameseNorm、Group-Layer Normalization 分别用于简化归一化、对齐异构 Token 分布以及保持不同语义空间。Inter-layer Residual、Mixing-and-Reverting 和辅助损失则用于缓解梯度衰减、表示退化和稀疏路由不均衡等问题。

4）线上执行 / 有效 Scaling。 结构可堆叠并不等于能够在线扩展。模型还需要通过规则大算子、序列压缩、用户侧复用、KV Cache、稀疏激活和算子融合提高硬件利用率，并控制 RT 与成本。只有新增参数和 FLOPs 能够被高效执行并带来效果收益，模型才真正实现了 Scaling。

4.3 两个问题形成的共同演进方向

从这些代表工作可以看到，当前搜推模型 Scaling 已经形成两条相对清晰的主线：

  1. 输入侧：从特征拼接走向结构化 Tokenization。 核心不是简单增加 Token 数，而是在业务语义、信息粒度和计算成本之间取得平衡，把用户、物品、Query、上下文和行为序列组织为统一的 T×DT\times DT×D 表示。
  2. 主干侧：从碎片化模块走向统一的 Transformer-like Backbone。 Self-Attention 或其替代结构负责跨 Token 交互，FFN/MoE 负责扩展通道容量，Norm/Residual 保证深层训练稳定，系统优化负责把模型规模转化为可承受的线上计算。

因此，Scaling Up 不能只看总参数量。评估一个方案是否真正实现了有效扩展，还需要同时观察 Dense 参数、Sparse 参数、激活参数、FLOPs、Token 数、序列长度、MFU、线上 RT、成本和业务收益。最终目标不是得到一个更大的模型，而是让增加的参数和计算能够持续转化为更好的排序效果。

4.4 实践中需要特别注意的问题

在具体落地时，验证新范式的结构有效性只是第一步。模型从“能够训练”走向“规模增加后仍然持续有效”，还需要同时解决数据供给、参数收敛、热启动、训练稳定性和线上执行等问题。

1）参数规模扩大后，需要同步增加有效样本量和训练步数。 模型容量增加并不会自动带来收益，新增参数只有获得足够的训练信号才能收敛。如果仍使用相同的数据窗口、样本量和训练步数，大模型可能表现为收敛不足；反过来，如果在有限样本上反复训练，也容易转向记忆高频用户、物品和特征组合。实践中需要随着 Dense 参数、激活参数和 Token 数增长，同步评估样本规模、数据覆盖度、负样本数量、训练 Token 数和总训练 Compute，并通过训练集—验证集差距判断是欠拟合还是过拟合。

2）Multi-epoch 训练时，需要特别处理热加载的 Embedding。 为了让新增的 Dense 主干充分收敛，可以在同一批样本上训练多个 Epoch。但搜推模型通常会从线上或上一版本热加载大规模 Embedding，这些参数已经携带较强的历史记忆；如果继续在同一批样本上反复更新，尤其是高频 ID 的 Embedding，容易比主干更早过拟合。

一种可用的工程技巧是：Dense 主干在多个 Epoch 之间连续训练，但每个 Epoch 开始前，都从同一个 Warm-start Checkpoint 重新加载 Embedding 参数。 这样，Dense 主干可以持续吸收多轮训练信号，而 Embedding 不会在相同样本上累计多轮更新。实现时还应同步重置 Embedding 对应的优化器状态，例如 Adam 的一阶、二阶动量，避免参数已经恢复而优化器状态仍保留上一 Epoch 的更新记忆。该方法并非所有任务都必然收益，需要结合 ID 频次分布、数据新鲜度和验证集表现进行判断。

3）可以通过 Global Token 实现更平滑的热启动。 当模型从旧的 Embedding + MLP 结构升级到 Token + Transformer-like 主干时，新旧模型的中间表示通常无法直接一一对应。可以将旧模型已经收敛的高层表示或输出表示经过线性投影，作为一个样本相关的 Global Token 加入新模型的 Token 序列。训练初期，新主干可以通过 Global Token 复用旧模型的稳定信息，再逐步学习其他 Token 带来的增量能力，相当于为结构升级提供一条信息捷径。如果新旧模型都包含可学习的 Global Token，也可以直接从旧 Checkpoint 加载该参数，但样本相关的旧模型表征通常能提供更强的热启动信号。

4）新参数与热加载参数需要采用不同的优化策略。 新增的 Tokenizer、Transformer-like Block、FFN 或 MoE 参数从随机初始化开始，而 Embedding 和旧模型模块已经处于相对稳定的区域，两者使用完全相同的学习率容易顾此失彼。实践中可以使用分组学习率：新参数采用较大的学习率和 Warmup，热加载参数采用较小学习率；同时配合 Norm、Residual、梯度裁剪以及必要的辅助损失，观察各层梯度、激活分布和专家负载，确保新增深度与容量真正得到训练。

5）需要验证新增容量是否能够转化为线上有效计算。 离线效果提升并不代表具备上线价值。排序模型的计算还会被候选数量放大，因此需要同时观察 P50/P99 RT、QPS、显存、通信开销、MFU、单候选 FLOPs 和激活参数，而不能只报告总参数量。优先将计算组织为规则大矩阵，并结合用户侧表示复用、Cache、序列压缩、稀疏激活和算子融合，把更多时间用于有效计算，而不是消耗在大量小算子、数据搬运和动态调度上。

6）结构升级需要分阶段验证，并保留回滚路径。 建议先验证单一模块替换和离线 Scaling 趋势，再进行小流量灰度，最后逐步扩大流量。除了主指标，还需要关注分用户、分场景、分频次和冷启动样本的收益与退化，并检查训练—推理特征一致性、模型稳定性和资源水位。只有效果、RT、成本和稳定性同时满足要求，才能认为新的 Scaling 范式在业务中真正成立。
